MLX AudioGen runs four on-device model families plus one opt-in cloud model. **On-device:** MusicGen for autoregressive text-to-audio, Stable Audio Open for diffusion-based stereo generation, MOSS-SoundEffect for text-to-sound-effect generation, and Demucs v4 for stem separation — all running locally on your Mac's Metal GPU. **Cloud:** Lyria, Google's hosted music model on Vertex AI, is the first non-local model and is off by default. Each on-device family has multiple variants tuned for different output quality, speed, and conditioning capabilities. Pre-converted MLX weights for all on-device variants are published on [HuggingFace](https://huggingface.co/jasonvassallo) and download automatically on first use.

<Tip>
If you're using MLX AudioGen for the first time, start with a small model. The `musicgen-small` and `stable-audio-open-small` variants download and run fastest, and they produce good results for most prompts. Switch to larger variants once you're comfortable with the generation workflow.
</Tip>

## MusicGen

MusicGen is an autoregressive model from Meta that generates audio by predicting audio codec tokens from a text prompt. It encodes your prompt with T5, runs an autoregressive transformer with classifier-free guidance, then decodes tokens back to audio using EnCodec. All MusicGen variants output at 32 kHz.

### Variants

| Model | Size variants | Output | Conditioning |
|---|---|---|---|
| MusicGen | small, medium, large | Mono | Text |
| MusicGen Stereo | small, medium, large | Stereo | Text |
| MusicGen Melody | base, large | Mono | Text + melody audio |
| MusicGen Stereo Melody | base, large | Stereo | Text + melody audio |
| MusicGen Style | base | Mono | Text + style audio |

### Choosing a MusicGen variant

- **small** — fastest generation, lowest memory use, good for prototyping and short clips
- **medium** — balanced quality and speed, suitable for most production uses
- **large** — best quality, slower generation, highest memory use
- **stereo variants** — use 8 codebooks instead of 4 for left/right channel separation; useful when output will be played back on stereo systems
- **melody variants** — add `--melody my_audio.wav` to condition generation on the pitch contour of a reference file; works without `--melody` for text-only generation
- **style variants** — add `--style-audio reference.wav` to match the timbre and texture of a reference clip using MERT feature extraction; `--style-coef` controls how strongly the text prompt influences the output (default: 5.0)

## Stable Audio Open

Stable Audio Open is a diffusion-based model from Stability AI that generates stereo audio at 44.1 kHz. It encodes your prompt with T5, applies rectified flow ODE sampling through a Diffusion Transformer (DiT), then decodes with an Oobleck VAE. The diffusion approach lets you control generation quality vs. speed through the number of steps and CFG scale.

### Variants

| Variant | Sample rate | Output | Notes |
|---|---|---|---|
| small | 44.1 kHz | Stereo | Open access, downloads automatically |
| 1.0 | 44.1 kHz | Stereo | Larger DiT, dual time conditioning — gated model |

### HuggingFace repo IDs

| Variant | HuggingFace repo |
|---|---|
| Stable Audio Open small | `stabilityai/stable-audio-open-small` |
| Stable Audio Open 1.0 | `stabilityai/stable-audio-open-1.0` |

<Warning>
Stable Audio Open 1.0 is a gated model. Before you can download or convert the weights, you must accept the license agreement on the [HuggingFace model page](https://huggingface.co/stabilityai/stable-audio-open-1.0) and authenticate with `huggingface-cli login`. The small variant has no gate and downloads automatically.
</Warning>

### Stable Audio parameters

Stable Audio uses different generation parameters than MusicGen:

| Parameter | Default | Description |
|---|---|---|
| `--steps` | 8 | Number of diffusion steps — more steps = higher quality, slower generation |
| `--cfg-scale` | 6.0 | CFG guidance scale — higher values follow the prompt more strictly |
| `--sampler` | `euler` | ODE sampler — `euler` or `rk4` |
| `--negative-prompt` | `""` | Describe what you don't want in the output |

## MOSS-SoundEffect

MOSS-SoundEffect is a text-to-sound-effect model from OpenMOSS. Instead of music, it generates sound effects — impacts, ambiences, foley, whooshes, and other non-musical audio — from a text prompt. It pairs a Qwen3-8B autoregressive language-model backbone with a 16-codebook RVQ audio tokenizer (delay-pattern decoding) and outputs mono audio at 24 kHz. The weights are quantized to 4-bit and licensed under Apache-2.0.

<Note>
MOSS-SoundEffect is available through the [web UI](/integrations/web-ui) and the [HTTP API](/api/generation) (`model: "moss_sfx"`). It is not exposed by the `mlx-audiogen` command-line tool, which generates with MusicGen and Stable Audio only.
</Note>

### Variants

| Variant | HuggingFace repo | Output | License |
|---|---|---|---|
| moss-sfx (4-bit) | `mlx-community/MOSS-SoundEffect-MLX-4bit` | 24 kHz mono | Apache-2.0 |

MOSS-SoundEffect runs through the [`mlx-speech`](https://pypi.org/project/mlx-speech/) package, which auto-resolves both the model weights and its companion `appautomaton/openmoss-audio-tokenizer-mlx` codec from the HuggingFace cache on first use. Because the weights resolve on demand, the model is always offered in the web UI's model selector even before a local copy has been downloaded.

### MOSS-SoundEffect parameters

MOSS-SoundEffect uses autoregressive sampling parameters. In the web UI, selecting the MOSS-SoundEffect model exposes these sliders:

| Parameter | Default | Range | Description |
|---|---|---|---|
| Temperature | 1.5 | 0.1–2.0 | Sampling temperature — higher values produce more varied output |
| Top K | 50 | 1–500 | Number of top-k token candidates considered at each step |
| Top P | 0.6 | 0.05–1.0 | Nucleus-sampling probability mass |
| Repetition Penalty | 1.2 | 1.0–2.0 | Penalizes repeated tokens to avoid looping artifacts |

<Note>
These defaults are tuned for MOSS-SoundEffect and differ from MusicGen's. The web UI seeds them automatically when you select the model. Over the HTTP API, `top_p` and `repetition_penalty` default to MOSS-appropriate values, and when you omit `temperature`/`top_k` the server substitutes the MOSS defaults (1.5 / 50) in place of the MusicGen schema defaults. See the [generation API reference](/api/generation) for the exact wire-format defaults.
</Note>

## Lyria (cloud — Google Vertex AI)

Lyria is Google's hosted music-generation model, accessed through Vertex AI. It is the **first and only cloud model** in MLX AudioGen — unlike every other model on this page, it does **not** run on your Mac. Your prompt is sent to Google Cloud, generation happens on Google's infrastructure, and the finished audio is returned. Lyria is **opt-in and off by default**: until you configure it, it never appears in the model selector or the API model list, and the rest of MLX AudioGen stays fully local and offline.

<Warning>
**Lyria sends your prompt to Google Cloud.** Every other MLX AudioGen model runs entirely on-device with no data leaving your machine. Lyria is different — when you generate with it, your prompt text is transmitted to Google's Vertex AI and the audio is produced in the cloud. Only enable Lyria if you are comfortable with that egress. Generated audio is also **SynthID-watermarked** by Google (an inaudible watermark identifying it as AI-generated).
</Warning>

### Variants

| Model id | Status | Endpoint | Output |
|---|---|---|---|
| `lyria-002` | Stable | Vertex AI publisher `:predict` | ~30 s, 48 kHz stereo WAV |
| `lyria-3-pro-preview` | Preview | Vertex AI `/interactions` | Lyria 3 Pro — text input only |
| `lyria-3-clip-preview` | Preview | Vertex AI `/interactions` | Lyria 3 Clip — text input only |

<Note>
**`lyria-002` is Lyria 2 — the stable, verified model.** It returns a ~30-second 48 kHz stereo clip and supports an optional negative prompt and seed. The **`lyria-3-*` models are Lyria 3 in preview** — treat them as experimental: they use a different (newer) Vertex AI endpoint and accept only a text prompt (no negative prompt or seed). For dependable results, prefer `lyria-002`.
</Note>

### Enabling Lyria

Lyria requires the optional `[cloud]` extra and a configured Google Cloud project. **There is no API key** — authentication uses Google [Application Default Credentials (ADC)](https://cloud.google.com/docs/authentication/application-default-credentials), so it transparently uses whatever gcloud / Workload Identity Federation login is already set up on the host.

1. Install the cloud extra:

   ```bash
   pip install "mlx-audiogen[cloud]"
   ```

2. Authenticate with Application Default Credentials (one of):

   ```bash
   # Local development — opens a browser login
   gcloud auth application-default login
   ```

   On a server you would instead use a service account or Workload Identity Federation; ADC picks up whichever is configured.

3. Point MLX AudioGen at your project (and, optionally, a region):

   ```bash
   export GOOGLE_CLOUD_PROJECT="my-gcp-project-id"
   export GOOGLE_CLOUD_LOCATION="us-central1"   # optional; defaults to us-central1
   ```

When `GOOGLE_CLOUD_PROJECT` is set, the Lyria models appear in the web UI's model selector (badged as cloud) and in `GET /api/models`. When it is unset, Lyria is simply not offered and nothing else changes. `GOOGLE_CLOUD_LOCATION` only affects the `lyria-002` `:predict` endpoint; the Lyria 3 preview endpoint always uses Google's `global` location.

<Note>
Lyria runs as an ordinary `/api/generate` job — it is a single blocking cloud round-trip (batch), not a streaming connection. Generate with it the same way you generate with a local model; see the [generation API reference](/api/generation) for the request fields (`negative_prompt` and `seed` apply to `lyria-002`).
</Note>

## Demucs v4

Demucs v4 (HTDemucs) is Meta's hybrid source separation model. It takes a stereo audio file and splits it into separate stems. Unlike the generation models, Demucs does not take a text prompt — it analyzes the audio signal directly.

### Variants

| Variant | Stems | Output | Description |
|---|---|---|---|
| `htdemucs` | 4 (drums, bass, vocals, other) | 44.1 kHz stereo | Standard separation |
| `htdemucs_6s` | 6 (drums, bass, vocals, other, guitar, piano) | 44.1 kHz stereo | Adds guitar and piano stems |

Demucs uses a hybrid architecture combining a spectral U-Net and a temporal U-Net connected by a CrossTransformer. Pre-converted MLX weights auto-download from [HuggingFace](https://huggingface.co/jasonvassallo/demucs-htdemucs-mlx) when you first run separation.

## Picking the right model

<AccordionGroup>
  <Accordion title="I want to generate music from a text description">
    Start with **MusicGen small** for fast results. Use **MusicGen large** or **MusicGen Stereo large** when you need higher quality or stereo output for production use.
  </Accordion>
  <Accordion title="I want stereo output at CD quality (44.1 kHz)">
    Use **Stable Audio Open small** or **Stable Audio Open 1.0**. Stable Audio always outputs stereo at 44.1 kHz, compared to MusicGen's 32 kHz.
  </Accordion>
  <Accordion title="I want to guide generation with an existing melody or hummed idea">
    Use **MusicGen Melody** (base or large). Pass your reference audio with `--melody my_melody.wav`. The pipeline extracts the pitch contour from your file and uses it to condition generation alongside your text prompt.
  </Accordion>
  <Accordion title="I want to match the timbre or texture of a reference track">
    Use **MusicGen Style**. Pass your reference audio with `--style-audio reference.wav`. MERT extracts timbral features from the reference and blends them with your text prompt using dual classifier-free guidance.
  </Accordion>
  <Accordion title="I want to generate a sound effect, not music">
    Use **MOSS-SoundEffect**. Describe the sound — for example "heavy wooden door slam" or "distant thunder rumble" — and it generates a 24 kHz mono effect. It's available in the web UI and over the HTTP API (`model: "moss_sfx"`), not the CLI.
  </Accordion>
  <Accordion title="I want to split a mixed track into individual stems">
    Use **HTDemucs** for 4-stem separation (drums, bass, vocals, other), or **htdemucs_6s** if you also need guitar and piano isolated separately.
  </Accordion>
  <Accordion title="I want Google's Lyria model and I'm OK sending prompts to the cloud">
    Use **Lyria** (`lyria-002`). It runs on Google Vertex AI rather than on your Mac, so your prompt is sent to Google Cloud and the audio is SynthID-watermarked. Enable it with the `[cloud]` extra plus a `GOOGLE_CLOUD_PROJECT` and Application Default Credentials — no API key. The `lyria-3-*` variants are preview/experimental. See [Lyria](#lyria-cloud-google-vertex-ai) above.
  </Accordion>
</AccordionGroup>
